# コミットメッセージ作成・コミット実行

**ステージング済みファイルの差分のみ**を参照してコミットメッセージを作成し、`git commit` まで実行する。

## 厳守事項（必ず従うこと）

- **使用する Git コマンドは `git diff --staged` のみ**
- `git diff HEAD` や未ステージの変更は参照禁止
- チャット履歴や会話内容は一切参照禁止
- 上記に反する場合は、まず「ステージングされている内容のみを参照します」と明言してから実行する

## 手順

1. **差分の取得**
   - `git diff --staged` のみを実行してステージング済みの変更を取得
   - **ステージング済みの差分が無い場合は、「ステージングされているファイルがありません」と伝え、メッセージ作成・コミット実行は行わない**

2. **差分の分析**
   - 変更されたファイル、追加・削除された行、変更の性質を客観的に分析
   - ステージング済みの内容のみを情報源とする

3. **コミットメッセージの作成**
   - 以下の形式で記述する：

   ```
   <type>: <subject>

   <body>
   ```

4. **コミットの実行**
   - 作成したメッセージで `git commit -m "type: subject" -m "body"` を実行
   - 複数行の body は `-m` を複数指定するか、1つの `-m` に改行含めて渡す
   - コミット完了後、`git log -1 --oneline` でコミット結果を確認して報告する

## type の種類

- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの意味に影響を与えない変更（空白、フォーマット、セミコロンなど）
- `refactor`: バグ修正や機能追加ではないコードの変更
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツールの変更、ライブラリの更新など

## subject のルール

- 日本語で「〜を追加」「〜を修正」などの形式
- 50文字以内
- 文末に句読点を付けない

## body のフォーマット

```
【変更内容】
- 具体的な変更点を箇条書きで記述
- 複数の変更がある場合は項目ごとに記述
```

- 72文字以内で折り返し
- 変更内容は簡潔に記述

## 補足

- ステージング済み差分（`git diff --staged`）から読み取れる内容のみに基づいて type・subject・body を作成すること
- チャット履歴や会話の文脈は完全に無視すること
- 未ステージの変更はコミット対象外のため、参照してはならない
